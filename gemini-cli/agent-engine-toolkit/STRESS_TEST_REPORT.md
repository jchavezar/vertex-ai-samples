## 🛠️ Phase 1: Deployment & Architecture Evolution

### 🏁 Test 1: Legacy CLI Deployment (`to_agent_engine`)
- **Status**: **FAILED**.
- **Root Cause**: Dependency conflict between workspace `google-adk` (v1.15.1) and system requirements (v1.25.1+).
- **Solution**: Moved to a standalone `uv` environment to isolate modern SDK dependencies.

### 🏁 Test 2: Pydantic Validation & Naming
- **Status**: **FAILED**.
- **Root Cause**: `LlmAgent` name validation error. Hyphens (`-`) are not permitted in agent identifiers.
- **Solution**: Renamed agent to `root_agent`.

### 🏁 Test 3: Standardizing on `vertexai.Client`
- **Status**: **SUCCESS**.
- **Discovery**: Objects returned by `client.agent_engines.list()` are metadata-only. One must call `client.agent_engines.get()` to retrieve a callable instance.

## ⚠️ Observed Breaking Points & Evidence

### 1. Hard Quota Limit (429)
The primary failure point is the administrative quota:
`Quota exceeded for quota metric 'Query Reasoning Engine requests' and limit '90 per minute per region'`.

### 2. Double-Hit Capacity Loss
Because each unique session requires a creation call, the effective user capacity is exactly **Quota / 2**. firing 80 users generates 160 calls, exceeding the 90 RPM limit.

---

## 💡 Recommendations for Scale
1. **Session Reuse**: Avoid `async_create_session` for every query. Reusing sessions increases capacity from **30 to 90 concurrent queries per minute**.
2. **Quota Increase**: To support 80+ concurrent unique users, the project quota for `aiplatform.googleapis.com/reasoning_engine_service_query_requests` must be increased to **200 RPM**.

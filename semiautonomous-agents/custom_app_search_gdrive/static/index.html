1
AUTH🌐 Browser→🔐 Google Auth
User clicks "Sign in with Google"
Google Identity Services opens a popup consent screen. The user authenticates and grants the cloud-platform scope.
2
AUTH🔐 Google Auth→🌐 Browser
Google returns ID Token + Access Token
Two tokens issued and held in memory only. The ID token is a signed JWT identifying the user. The access token is short-lived for calling GCP APIs.
ID Token: eyJhbGci… (JWT — contains email, name, sub) Access Token: ya29.a0ARrd… (OAuth2 — scope: cloud-platform) Expires in: 3600s
3
TOKEN🌐 Browser→⚙️ FastAPI
Query + both tokens POSTed to backend
On every search, the browser POSTs the query, ID token, and access token to /api/search over HTTPS. The backend never stores tokens.
POST /api/search { "query": "…", "credential": "<id_token>", "access_token": "<ya29…>" }
4
VERIFY⚙️ FastAPI→🔐 Google Auth
Backend verifies the ID token signature
Uses google-auth to verify the JWT cryptographic signature against Google's public keys. Confirms the token was issued for this app and is not expired.
google.oauth2.id_token.verify_oauth2_token(credential, request, CLIENT_ID) → { "email": "user@paramount.com", "sub": "…", "exp": … }
5
VERIFY⚙️ FastAPI→🔐 Google Auth
Backend validates the access token
Calls /oauth2/v3/tokeninfo to confirm the access token is valid and the email matches the ID token. Prevents token substitution attacks.
GET https://www.googleapis.com/oauth2/v3/tokeninfo?access_token=ya29… → { "email": "user@paramount.com", "scope": "…cloud-platform…" }
6
API⚙️ FastAPI→🔍 Vertex AI Search
Two parallel calls using the user's own token
Fires two Discovery Engine requests simultaneously — one fast (no summary, returns results immediately) and one for AI summary generation. Both use Authorization: Bearer <user_access_token>.
POST discoveryengine.googleapis.com/…/default_search:search Authorization: Bearer ya29… ← user's own token (NOT service account) { "query": "…", "userInfo": { "userId": "user@paramount.com" } }
7
ACL🔍 Vertex AI Search↺Google Drive ACL
Server-side ACL enforcement
Vertex AI Search checks Drive permissions server-side. Only files the user has read access to are returned — even if other files exist in the datastore. No client-side filtering.
8
STREAM⚙️ FastAPI→🌐 Browser
Results stream immediately — summary follows
File results arrive via SSE as soon as the fast call completes. The AI summary populates the summary card separately once generation finishes.
data: {"type":"results", "data": { results: […] }} data: {"type":"summary", "data": { summaryText: "…" }}
Zero service-account exposure on search
The Discovery Engine API call is made exclusively with the signed-in user's OAuth token. No service account key or environment credential is used. If the token is invalid, the search fails — it cannot fall back to elevated credentials. ACL enforcement is fully server-side at Google.
